# Feature Detail: Dependency Validation

**Feature ID**: NF04
**Name**: Dependency Validation
**Description**: Systematic tracking and validation of external command dependencies per script.
**Status**: Draft
**Created**: 2026-06-15
**Last Updated**: 2026-06-15

## Overview

Each library script depends on specific external commands (docker, kubectl, terraform, openssl, etc.), but there is no centralized dependency manifest or pre-flight validation mechanism. This feature proposes a systematic approach to declaring, tracking, and validating external dependencies so users get clear early feedback when required tools are missing.

## Latest Review

DFX gap analysis (Design for Maintainability / Design for Configuration). Functions use `have <cmd>` checks individually, but no aggregated dependency map exists.

## Key Changes

1. Define a per-script dependency declaration format (e.g., header comment or metadata variable)
2. Build a dependency audit script that scans all scripts and reports missing commands
3. Add a `cws_bash_check_deps` function for pre-flight validation of a specific domain
4. Generate a dependency matrix (script × required commands) for documentation

## Implementation Notes

- Existing `have` function provides the runtime check; this feature adds the declaration/audit layer.
- Must not add load-time overhead; dependency checking remains opt-in at function call time.
- The audit script should be runnable standalone (no need to source the full library).
- DFX Category: Design for Maintainability (DFM) / Design for Configuration (DFCfg).

## Future Evolution Suggestions

- Automatic installation suggestions for missing dependencies (via package manager detection).
- CI integration: fail pipeline if a script introduces an undeclared dependency.
- Version constraint support (e.g., "requires terraform >= 1.5").
