# UML 总论

> UML（Unified Modeling Language，统一建模语言）的定义、价值、14种图表分类与选择指南。

## 什么是 UML

UML（Unified Modeling Language，统一建模语言）是一种用于说明、可视化、构建和编写软件系统制品的开放方法。它是非专利的第三代建模和规约语言，由对象管理组织（OMG）于1997年发布1.0版本规范。

UML 提供了一套全球公认的建模标准，为软件开发人员提供了一种标准的可视化语言来表示软件系统的结构、行为和交互。它特别适用于对大规模、复杂系统进行建模，尤其是在软件架构方面展现了一系列最佳工程实践。

## 为什么要使用 UML

- **标准化沟通**：UML 图为团队提供了一种通用的可视化语言，消除理解歧义
- **系统设计**：在编码前可视化系统架构，发现设计问题
- **文档记录**：作为系统设计的正式文档，便于维护和传承
- **需求分析**：通过用例图等捕捉和验证用户需求
- **代码生成**：部分工具支持从 UML 模型生成代码框架

## UML 图表分类

UML 2.x 规范定义了 **14 种图表类型**，分为两大类：

### 结构图（Structural Diagrams）— 7 种

描述系统的静态结构，展示系统中的元素及其关系：

| 图表类型 | 英文名称 | 主要用途 |
|---------|---------|---------|
| 类图 | Class Diagram | 展示类、接口、属性、方法及它们之间的关系 |
| 对象图 | Object Diagram | 展示某一时刻系统中对象的实例及其关系 |
| 组件图 | Component Diagram | 展示系统中组件的组织结构和依赖关系 |
| 部署图 | Deployment Diagram | 展示硬件节点和软件组件的部署关系 |
| 包图 | Package Diagram | 展示模型元素的分组和组织结构 |
| 组合结构图 | Composite Structure Diagram | 展示类的内部结构和协作关系 |
| 轮廓图 | Profile Diagram | 用于扩展 UML 元模型的自定义构造型 |

### 行为图（Behavioral Diagrams）— 7 种

描述系统的动态行为，展示系统中元素如何交互和变化：

| 图表类型 | 英文名称 | 主要用途 |
|---------|---------|---------|
| 用例图 | Use Case Diagram | 展示系统与外部参与者的交互和功能需求 |
| 时序图 | Sequence Diagram | 展示对象间消息传递的时间顺序 |
| 活动图 | Activity Diagram | 展示业务流程或算法的控制流和数据流 |
| 状态机图 | State Machine Diagram | 展示对象在其生命周期中的状态变化 |
| 通信图 | Communication Diagram | 展示对象间的连接关系和消息传递 |
| 定时图 | Timing Diagram | 展示对象状态随时间变化的精确时序约束 |
| 交互概览图 | Interaction Overview Diagram | 结合活动图和时序图，展示复杂交互流程 |

## 常用 UML 图选择指南

| 我想表达... | 使用... | 关键词 |
|------------|--------|--------|
| 系统的静态结构（类、接口） | 类图 | 领域模型、类关系、API设计 |
| 对象间的交互顺序 | 时序图 | 调用流程、消息传递、接口协作 |
| 用户能看到什么功能 | 用例图 | 需求分析、系统边界、角色权限 |
| 业务流程或算法步骤 | 活动图 | 工作流、审批流程、并发处理 |
| 对象的状态变化 | 状态机图 | 订单状态、生命周期、协议状态 |
| 系统的物理部署 | 部署图 | 服务器拓扑、网络架构 |
| 组件间的依赖 | 组件图 | 微服务架构、模块划分 |

## 核心 UML 图速览

软件开发中使用频率最高的 5 种 UML 图表：

### 类图（Class Diagram）
面向对象设计的核心图表，展示类的属性、方法以及类之间的 6 种关系（依赖、关联、聚合、组合、泛化、实现）。适用于设计阶段定义领域模型、逆向工程理解代码结构、API 设计和文档化。

### 时序图（Sequence Diagram）
展示对象之间按时间顺序的消息交换。主要元素包括生命线（Lifeline）、激活条（Activation Bar）、消息（Message）和组合片段（Combined Fragment）。组合片段支持 `alt`（条件分支）、`loop`（循环）、`opt`（可选）、`par`（并行）等控制结构。适用于设计 API 调用流程、分析复杂业务逻辑的执行顺序。

### 用例图（Use Case Diagram）
从用户视角展示系统功能需求。核心元素有参与者（Actor）、用例（Use Case）、关联（Association）、包含（Include）和扩展（Extend）。适用于需求收集和分析、与利益相关者沟通系统功能。

### 活动图（Activity Diagram）
展示业务流程、工作流或算法的步骤和控制流。支持并行执行（Fork/Join）、泳道（Swimlane）划分职责、条件分支（Decision/Merge）。适用于业务流程建模、算法逻辑可视化、工作流设计。

### 状态机图（State Machine Diagram）
展示单个对象在其生命周期中响应事件的状态变化。包含状态（State）、转换（Transition）、事件（Event）、守卫条件（Guard）和动作（Action）。适用于订单状态管理、UI 组件状态设计、协议状态机建模。
