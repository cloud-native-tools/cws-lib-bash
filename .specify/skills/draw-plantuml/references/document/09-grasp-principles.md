# GRASP 九大原则

> GRASP（General Responsibility Assignment Software Patterns，通用职责分配软件模式）是 Craig Larman 在《Applying UML and Patterns》中提出的九个指导如何将职责分配给类和对象的基本原则。

## 核心思想

GRASP 不关心「怎么画类图」，而是回答一个更根本的问题：**这个职责应该由哪个类来承担？**

职责分配是面向对象设计的核心活动。设计的好坏，本质上取决于职责是否被合理地分配给了合适的类。

## 原则一：信息专家（Information Expert）

**核心问题**：哪个类应该拥有某个职责？

**原则**：将职责分配给拥有完成该职责所需信息的类。

这是面向对象设计中最基本、最常用的原则。如果一个类已经拥有完成某操作所需的数据，那么操作就应该放在这个类中。这自然地引导出「高内聚」的设计。

**反例**：把所有业务逻辑都写在一个 Service 类中，而数据对象只作为 DTO 传递——这就是典型的「贫血模型」，违背了信息专家原则。

## 原则二：创建者（Creator）

**核心问题**：谁应该负责创建某个类的实例？

**原则**：如果满足以下条件之一，类 B 应该负责创建类 A 的实例：
- B 包含或聚合了 A
- B 记录了 A
- B 密切使用 A
- B 拥有创建 A 所需的初始化数据

**实践意义**：这解释了为什么 `Order` 应该创建 `OrderItem`，而不是由一个外部的 `OrderFactory` 来创建——除非创建逻辑极其复杂。

## 原则三：控制器（Controller）

**核心问题**：谁应该处理来自 UI 层的系统操作请求？

**原则**：将处理系统事件的职责分配给以下两类对象之一：
- **外观控制器（Facade Controller）**：代表整个系统或子系统的类
- **用例控制器（Use-Case Controller）**：代表单个用例场景的类

**实践意义**：这就是 MVC 中 Controller 层的理论基础。控制器不应该包含业务逻辑，它只是一个「协调者」——接收请求，委托给领域对象处理。

## 原则四：低耦合（Low Coupling）

**核心问题**：如何减少变更的影响范围？

**原则**：尽可能降低类之间的依赖关系，使得一个类的变更不会波及太多其他类。

低耦合是评价设计质量的核心指标。GRASP 强调的不仅是「减少耦合」，而是「在满足功能需求的前提下，选择耦合度最低的方案」。

## 原则五：高内聚（High Cohesion）

**核心问题**：如何让类的职责聚焦？

**原则**：一个类应该只承担密切相关的一组职责。如果一个类做了太多不相关的事情，就应当拆分。

高内聚和低耦合相辅相成。一个高内聚的类自然对外部依赖更少，耦合度更低。

## 原则六：多态（Polymorphism）

**核心问题**：如何处理基于类型的变化行为？

**原则**：当相关行为因类型（类）而异时，使用多态将变化行为分配给行为所对应的类型，避免使用条件判断（if-else/switch）。

**实践意义**：这是策略模式、状态模式等设计模式的底层原则。替代类型码的条件逻辑，用继承和多态分发行为。

## 原则七：纯虚构（Pure Fabrication）

**核心问题**：当信息专家原则导致低内聚或高耦合时怎么办？

**原则**：创建一个人造的、不代表领域概念的类，将一组高内聚的职责分配给它。

**实践意义**：这就是为什么我们需要 Repository、Factory、Adapter 等类——它们不是领域模型中的真实概念，但能让设计更清晰。这是对信息专家原则的必要补充。

## 原则八：间接（Indirection）

**核心问题**：如何解耦两个直接耦合的组件？

**原则**：在两个类之间引入一个中间类，避免它们直接耦合。

**实践意义**：Adapter、Mediator、Facade 等模式都是间接原则的体现。在分层架构中，Service 层本身就是领域层和表现层之间的间接层。

## 原则九：防止变异（Protected Variations）

**核心问题**：如何让系统对变化有弹性？

**原则**：识别系统中可能发生变化的点，用接口或抽象类封装这些变化点，使变化的影响局限在接口之后。

这是 GRASP 中最高层次的原则，与开闭原则（OCP）一脉相承。核心思想是：**预测变化，隔离变化**。

## GRASP 与 GoF 设计模式的关系

GRASP 是比 GoF 设计模式更基础的原则。GoF 的 23 个设计模式本质上是对 GRASP 原则的具体实现。

| GRASP 原则 | 对应的 GoF 模式 |
|-----------|----------------|
| 多态 | Strategy、State、Command |
| 纯虚构 | Factory、Repository |
| 间接 | Adapter、Mediator、Facade |
| 防止变异 | Bridge、Decorator、Observer |
| 控制器 | Front Controller、Command |
