# UML 建模方法论与最佳实践

> 经典 UML 建模方法论精华，涵盖 Fowler 务实哲学、OMT 方法论、潘加宇建模工作流，以及敏捷开发中的 UML 实践。

## Martin Fowler 的务实建模哲学

来源：《UML Distilled》

### 三种 UML 使用模式

| 模式 | 目的 | 详细程度 | 工具 |
|------|------|----------|------|
| **草图模式（Sketch）** | 沟通和探索想法 | 非正式，手绘 | 白板、draw.io |
| **蓝图模式（Blueprint）** | 详细设计文档 | 完整、精确 | CASE 工具 |
| **编程语言模式** | 从模型生成代码 | 极其精确 | MDA 工具 |

**核心主张**：绝大多数情况下草图模式就足够了。UML 是沟通语言不是方法论，用 20% 就能覆盖 80% 的需求（类图、时序图、用例图）。

### Fowler 的建模原则

1. 先理解问题，再画图——不要在还没想清楚的时候就开始画 UML
2. 图和代码要同步——图不能反映真实代码就没有价值
3. 选择性地使用——简单场景用文字描述即可
4. UML 是团队的语言——团队内部约定比 UML 规范更重要

## OMT 方法论（对象建模技术）

来源：《UML面向对象建模与设计》— Blaha & Rumbaugh

### 三种模型视角

| 视角 | 关注点 | 核心 UML 图 |
|------|--------|------------|
| 对象模型（静态结构） | 「有什么」 | 类图、对象图 |
| 动态模型（时间行为） | 「何时发生」 | 状态图、时序图 |
| 功能模型（数据变换） | 「做什么」 | 用例图、活动图 |

三者不是独立的模型，而是同一系统的三个互补切面。

### OO 开发流程

```
系统构思 → 分析 → 系统设计 → 类设计 → 实现
```

## 潘加宇的建模工作流

来源：《软件方法》

### 核心公式

> **利润 = 需求 - 设计**

需求是价值来源，设计是成本来源。建模的目的是最大化需求、最小化设计。

### 四工作流

```
业务建模 → 需求 → 分析 → 设计
```

1. **业务建模**：理解组织业务流程（不是画用例图）
2. **需求**：系统为外部执行者提供的价值（系统用例）
3. **分析**：识别支撑需求的核心概念（分析类图/领域模型）
4. **设计**：将分析模型映射到技术方案

**核心原则**：用例不是功能分解，而是「执行者使用系统达成目标的过程」。

## 跨书共识

| 维度 | 共识 |
|------|------|
| 建模的本质 | 思维方式而非画图技术 |
| UML 定位 | 沟通工具而非目的 |
| 使用方式 | 草图模式优先 |
| 核心流程 | 理解业务 → 定义需求 → 分析领域 → 设计实现 |
| 迭代开发 | UML 图是活文档，随代码一起演进 |

## 何时使用/不使用 UML

### 推荐使用

- ✅ 复杂系统设计阶段，需要可视化架构
- ✅ 团队协作，需要统一的设计语言
- ✅ 遗留系统重构，需要理解现有结构
- ✅ 需求评审，需要用例图澄清功能边界
- ✅ 技术文档，需要长期维护的设计参考

### 不推荐使用

- ❌ 简单 CRUD 应用，代码即文档
- ❌ 快速原型开发，迭代速度优先
- ❌ 个人小项目，过度设计

## 绘图原则

1. **目的驱动**：明确画图的目的，避免为了画图而画图
2. **适度抽象**：只展示与当前讨论相关的细节
3. **一致性**：保持命名、符号、风格的一致性
4. **迭代细化**：从高层抽象开始，逐步细化
5. **与代码同步**：定期更新以反映代码变化

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 试图画出所有类和关系 | 只画关键类和核心关系 |
| 混淆聚合和组合 | 部分能否独立存在？能→聚合，不能→组合 |
| 过度使用 UML 特性 | 80%场景只需类图、时序图、用例图 |
| 画图后不维护 | 将 UML 纳入版本控制，与代码一起更新 |
| 用 UML 替代沟通 | UML 是辅助工具，不能替代面对面讨论 |

## 敏捷开发中的 UML

- **草图优先**：白板手绘快速表达想法，拍照存档
- **按需建模**：只为复杂模块绘制详细 UML
- **代码即模型**：利用 IDE 的反向工程功能从代码生成 UML
- **轻量级文档**：用 PlantUML/Mermaid 文本化工具，便于版本控制

## 推荐阅读路径

| 目标 | 推荐路径 |
|------|----------|
| 快速上手 | 先读《UML Distilled》，掌握核心 20% |
| 系统学习 OO 设计 | 精读《Applying UML and Patterns》，掌握 GRASP |
| 完整的建模方法论 | 读《软件方法》+《UML面向对象建模与设计》 |
| 从业务到代码全流程 | 按《软件方法》的四工作流实践 |
