# 架构制图方法论

## 概述

架构制图是将架构设计可视化表达的过程。好的架构图是架构设计的最佳载体——没有图或者画不好图，就无法有效传达设计意图。架构制图的本质目标是将制图从一门自由手艺变成一项科学工程：系统、严谨、完整、标准化，同时可重复、可持续和高效。

## 为什么需要架构图

架构图在软件项目中的核心作用：

- **理解对齐**：让项目组所有成员对整个系统的理解对齐，形成共识
- **工作量化**：有了清晰的架构，开发工作能被准确量化
- **标准术语**：定义和解释系统中涉及的关键概念，统一沟通语言
- **言之有物**：为技术讨论提供实物载体
- **知识沉淀**：作为文档资产持续沉淀，新人可快速上手

## 架构核心要素

软件系统的架构通常包含四类核心要素：

- **元素（Elements）**：模块、组件、结构体、子系统
- **关系（Relationships）**：交互、依赖、继承、组合、聚合
- **属性（Properties）**：名称、职责、接口、实现限制等
- **原理（Principles）**：拆分依据、设计原则、决策原因等

## 制图方法对比

### UML

UML 作为"统一建模语言"，包含14种图类型：

**结构图**：类图、组件图、部署图、包图、对象图、组合结构图、轮廓图
**行为图**：用例图、活动图、时序图、状态机图、通讯图、交互概览图、时间图

优势：
- 各种记号和语义有严谨定义，不会有歧义
- 世界范围内广泛使用的标准规范
- 沟通成本低

劣势：
- 过于复杂（14种图类型）
- 大部分人只熟悉20%左右的内容

### 4+1 视图模型

通过5种视图从不同角度描述架构：

- **逻辑视图（Logical View）**：系统为终端用户提供的功能（类图、状态图）
- **过程视图（Process View）**：系统的动态行为（时序图、活动图）
- **开发视图（Development View）**：程序员视角（组件图、包图）
- **物理视图（Physical View）**：系统部署拓扑（部署图）
- **场景（Scenarios）**：用例或场景（用例图）

核心思想：架构需要通过多种视图来描述，这些视图来源于不同项目干系人的视点。

### C4 模型（推荐）

"抽象优先"的架构制图方法，像 Google Map 一样通过不同层次的细节建立代码的导览图：

**Level 1 - System Context Diagram**
- 展示系统全貌的顶层大图
- 包含：中心软件系统、周边用户、其他交互系统
- 受众最广，技术和非技术人员都能理解

**Level 2 - Container Diagram**
- 将系统拆分为容器（独立部署单元）
- 容器 = 单个应用或数据存储（微服务、SPA、数据库等）
- 包含主要技术选型和通讯方式

**Level 3 - Component Diagram**
- 将容器进一步拆分为组件
- 组件 = 一组通过良好接口定义封装在一起的相关功能
- 适用于架构师和开发者

**Level 4 - Code（可选）**
- UML类图或E/R图级别
- 通常只在非常重要和复杂的组件才需要
- 优先考虑自动化生成

**补充图**：
- 系统全景图（System Landscape）：企业级全部系统
- 动态图（Dynamic）：关键链路的运行时行为
- 部署图（Deployment）：逻辑节点与物理节点的映射

### arc42

架构文档模板，与 C4 模型兼容：

- 第3章 Context = C4 L1
- 第5章 Building Block View = C4 自顶向下拆分
- 第6章 Runtime View = C4 动态图
- 第7章 Deployment View = C4 部署图

## 架构制图五环方法论

### 1. 理解制图目标

- **准确（Accurate）**：错的图比没有图还糟糕
- **完整（Complete）**：覆盖架构核心要素和关键信息
- **清晰（Clear）**：带上图例，必要时加文字标注
- **一致（Consistent）**：同类型的图使用相同记号风格
- **简洁（Concise）**：用最少笔墨表达最核心内容

### 2. 找准受众和关注点

| 受众 | 关注点 |
|------|--------|
| 研发 | 技术选型、实现可行性、可维护性 |
| 运维 | 物理部署、网络连通性、可运维性 |
| 安全 | 安全风险、注入漏洞、权限问题 |
| 产品 | 项目进度、功能范围 |

### 3. 自顶向下逐层描述

核心原理：
- **分而治之**：层次化拆分，将系统按粒度逐级拆分
- **金字塔原理**：先说重点，自顶向下先抛主观点再用子观点论证

### 4. 使用多种架构视图

软件系统是多维立体的，单一视图不可能覆盖所有关键信息。不同架构视图有不同侧重点，相互正交且可配合使用。

### 5. 遵循规范和最佳实践

将架构制图从手艺变成工程：
- 遵循明确的规范进行约束和指引
- 遵循业界最佳实践持续提升技能

## 制图工具推荐

### draw.io
- 开源在线绘图软件
- 支持三方插件（如 c4-draw.io）
- 推荐使用离线桌面版

### PlantUML
- 文本制图（Diagram as Code）
- 轻量、高效、版本化、自动化、一致性
- 支持各种 UML 图和 C4 扩展
- 语雀文档支持内嵌

## 架构设计原则速查

### SOLID 原则

- **单一职责**：Do one thing and do it well
- **开闭原则**：用扩展取代修改
- **里氏替换**：父类能出现的地方子类一定能出现
- **接口隔离**：最小化组件之间的接口依赖
- **依赖反转**：依赖抽象而不是具体实现

### 通用原则

- **正交性**：各组件职责独立，边界清晰
- **高内聚**：同一组件内部是不可分割的整体
- **低耦合**：减少组件间耦合，增强可复用性
- **隔离变化**：将可能变化的部分隔离，减少影响范围
